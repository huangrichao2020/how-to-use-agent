# 示例 02：先架构后代码

[English](02-architecture-first.md) · [简体中文](02-architecture-first.zh-CN.md)

当 agent 意识到这不是普通 patch，而是更大的架构变化时，用这个模式。

## 原始 prompt

```text
这简直是大版本改动了，一条一条来，你先存档为大版本架构设计手册，然后我们一条条讨论和修改
```

## 开发者意图

这句话让 agent 慢下来，并把想法沉淀成一个持久的架构 artifact。

没有这一步，agent 很可能在设计还没稳定时就开始改代码。

## 可复用版本

```text
这是一次大版本架构变化。

先不要实现。

先写一份架构设计手册，并保存到项目 docs/wiki。
手册需要包括：
- 当前问题
- 目标架构
- 迁移阶段
- 风险
- 回滚策略
- 当前必须保持不变的东西

归档后，总结各章节，我们再逐条讨论。
```

## 验收检查

实现开始前，确认：

- 设计有稳定路径。
- 迁移被拆成阶段。
- 第一阶段足够小，可以测试。
- 回滚路径已经写清楚。
- 后续 agent 不需要读聊天记录也能找到设计。
