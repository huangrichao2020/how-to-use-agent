# 示例 03：渐进式迁移

[English](03-progressive-rollout.md) · [简体中文](03-progressive-rollout.zh-CN.md)

当 agent 想立刻切换到新架构时，用这个模式。

## 原始 prompts

```text
方案A 肯定是要做的，方案 1 直接做的话 有什么风险和收益？
```

```text
就是说渐进运行一段时间后，再全面切换生效会更好？
```

```text
那这期间是不是得停掉对记忆系统的改造了，聚焦这一件事
```

## 开发者意图

这些 prompt 把三个概念拆开：

- 方向已经被认可
- 上线策略还没有被认可
- 迁移期间应该暂停相邻高风险工作

## 可复用版本

```text
我同意目标方向。

在直接全量切换前，先比较：
- 直接全量切换
- 渐进式迁移
- shadow mode / 并行运行

对每个选项列出：
- 收益
- 失败模式
- 如何判断它正在工作
- 回滚成本

如果渐进式迁移更安全，迁移期间冻结相邻子系统。
除非迁移必须触碰它们，否则不要改。
```

## 好的 agent 行为

当出现以下情况时，agent 应该推荐分阶段路径：

- 改动触及 memory、tools、routing、startup 或 persistence
- 新旧行为可以并行一段时间
- 失败后用户很难定位原因
- 回滚需要知道哪些状态由哪个版本生成
